# AI作業場 実装指示書

本書は、マルチAIオーケストレーター・デスクトップアプリ「AI作業場(仮称)」の実装をAIコーディングエージェントに依頼するための仕様書である。設計判断は既に完了しており、本書の内容を正として実装すること。

> **実装AIへの注意**
> - 本書内の[確認が必要な点]は、実装時に最新の公式ドキュメント等での確認が必須の項目である。推測で埋めないこと。
> - 本書にない仕様の追加・変更が必要になった場合は、勝手に判断せず発注者(ユーザー)に確認すること。

---

## 1. コンセプト

複数のAI(Claude / GPT / Gemini)に「役割」を与え、1つのチャット欄から使い分け・連携させる個人用デスクトップアプリ。加えて、ローカルフォルダ内の資料から締め切りを自動検知するタスク管理機能を持つ。

- 利用形態:個人利用(配布予定なし)。API利用は各社公式APIをユーザー自身のキーで叩くBYOK方式
- 利用環境:ユーザー所有のPC 2台(自宅・持ち運び用)。クラウドストレージの同期フォルダ経由でデータ共有
- 主用途:小説執筆ワークフロー(執筆/整合性チェック/校正の分担)、学業タスク管理

## 2. 技術スタック

- デスクトップアプリ:Tauri + React を第一候補とする(軽量、ローカルファイルアクセス、OS通知、Rust側でのフォルダ監視が安定しているため)。実装AI側でElectronの方が確実に実装できる場合はElectron + Reactでも可
- データ保存:ローカルJSON/Markdownファイル(後述のワークスペース構造)。DBは使用しない(クラウドフォルダ同期に乗せるため)
- APIキー保存:OSキーチェーン(Windows資格情報マネージャー等)。同期フォルダには絶対に置かない

## 3. 接続するAI

| プロバイダ | API |
|---|---|
| Anthropic (Claude) | Messages API |
| OpenAI (GPT) | 公式API |
| Google (Gemini) | 公式API |

- モデル一覧はハードコードせず、各社の**モデル一覧取得エンドポイントから起動時に動的取得**してドロップダウンを生成する(新モデル対応・廃止モデル除去が自動化されるため)
- [確認が必要な点] 各社のモデル一覧エンドポイントのレスポンス形式、チャット非対応モデル(埋め込み用等)のフィルタ方法、トークン使用量(usage)の返却形式は、実装時に各社最新ドキュメントで確認すること
- [確認が必要な点] 各社APIの利用規約(個人アプリでの統合利用、出力の扱い)は実装前に最新版を確認すること

## 4. 機能仕様

### 4.1 役割カード(コア機能)

AIに役割を与える単位。ユーザーが無制限に作成・編集・削除できる。

- 構成:名前 / プロバイダ / モデル / システムプロンプト / 参照する共有資料 / ツール許可 / 自動起動トリガー(データ定義は§6参照)
- チャットでは `@役割名` のメンションで宛先カードを指定する
- カードの `color` フィールドはデータとして保持するが、**UI上の色分け表示は既定でオフ**(§5のデザイン方針を参照)

### 4.2 モデル切り替え(2段階)

1. カードレベル:カード編集でモデルを恒久変更
2. メッセージレベル:入力欄横のドロップダウンで「この送信のみ」モデルを上書き(`model_override`)

同一プロバイダ内の切り替え(例:Geminiの上位モデル→軽量モデル)も同じUIで行う。

### 4.3 共有バイブル(共有メモリ)

全カードが参照できる資料置き場(小説の制作バイブル等)。Markdownで保存し、カードの `shared_memory_refs` に登録された資料をAPI呼び出し時にシステムプロンプトへ注入する。

### 4.4 パイプライン(直列実行)

複数カードを順に実行するワークフロー(例:執筆→矛盾チェック→校正)。前段の出力を次段の入力とし、1ボタンで実行できる。定義は保存・再利用可能。

### 4.5 会話分岐(ブランチ)

- メッセージはツリー構造で保存(`parent_id` 方式、§6参照)。**フェーズ1の時点からデータはツリー構造にすること**(後からの改修は大規模になるため)。フェーズ1では分岐UIは作らない
- 分岐UI(フェーズ2):任意のメッセージから別ルートを作成、横並び比較、「Aを本線にする」で本線切り替え(`active_leaf_id` の付け替えのみで実現)
- 複数モデル比較:同一プロンプトを複数カードに同時送信し、横並び表示

### 4.6 締め切り検知(秘書役)

1. ユーザー指定フォルダを定期スキャン(間隔は設定可能、既定30分)
2. 新規・更新されたPDF / PPTX / DOCXからテキスト抽出
3. AIに「日付+行動を要する記述」をJSONで抽出させる(日付・タスク内容・確信度・元ファイル名・該当箇所の引用を返させる)
4. **確認UI必須**:検知結果は「これは締め切り?」カードとして表示し、ユーザーが「追加/無視」を選ぶまでタスク登録しない(誤検知の自動登録は信頼を損なうため)
5. 追加されたタスクはホーム画面右下のタスク欄に常駐表示

### 4.7 試験対策パイプライン

締め切り検知と同じフォルダスキャン基盤を使い、授業資料から「要点まとめ→予想問題→解説」を生成する定義済みパイプライン。

### 4.8 朝ブリーフィング

アプリ起動時(または1日の最初の起動時)、秘書役カードが「タスク一覧+締め切りが近い順+着手推奨コメント」を生成しホーム画面上部に表示する。

### 4.9 情報収集カード(Web取得)

ツール許可 `web_fetch: true` のカードは、ユーザーが明示的に渡したURLのページ内容を取得して分析できる。

- **取得はユーザーが明示したURLのみ**。クロール・巡回機能は実装しない
- 取得はアプリ側で1回のみ行い、結果を複数カードに配布する(同一ページの重複取得を避けるため)
- [確認が必要な点] 対象サイトの利用規約・robots.txtにより取得が禁止されている場合がある。実装ではrobots.txtの尊重を推奨

### 4.10 役割仕分けウィザード

- 入力:ユーザーが「やりたいこと」を自由文で記述
- 処理:接続済み全プロバイダに同一プロンプトで「タスクを役割に分割し、各役割の適任モデルを理由付きで提案せよ」と送信
- 表示:3社の提案を縦に並べて表示(各カードにプロバイダ名の小バッジ)
- 確定:ユーザーが役割ごとに担当モデルを選択→役割カードを自動生成(システムプロンプトの下書き付き)
- 初回起動時だけでなく、いつでも「新しい役割の相談」として起動可能

### 4.11 コストダッシュボード

- トークン使用量は**フェーズ1から記録**する(後から遡って集められないため)
- 表示:カード別・日別の使用量グラフ、概算コスト
- 単価はハードコードせず、設定画面でユーザーが入力・更新する(価格改定のたびのアプリ更新を避けるため)

### 4.12 テーマ・背景カスタマイズ

- ライト/ダークの2モード。CSS変数(デザイントークン)方式で実装し、`<body>` 属性1つで一括切り替え(将来のテーマ追加が変数セット1組の追加で済むため)
- 背景:なし/静止画/動画/AI生成の4種
  - 3層レイヤー構造:背景(最背面)→調光レイヤー(半透明幕。ライト時は白幕・ダーク時は黒幕を自動選択、濃度0〜80%をスライダー調整)→UIパネル(半透明+backdrop-filterぼかし)
  - 動画背景:バッテリー駆動時は静止画フォールバック、非アクティブ時は一時停止のオプションを設ける
  - AI生成背景:静止画は画像生成API([確認が必要な点] Anthropic APIに画像生成はないため、OpenAI/GoogleのAPIを使用。対応状況は実装時に確認)、動く背景はAIにCSS/Canvasアニメーションのコードを生成させる方式とする(動画生成APIはコスト・速度面で不向きなため)

### 4.13 通知(拡張ポイント)

- 実装するのはOSネイティブ通知のみ
- ただし内部は `NotificationSink` インターフェース経由とし、将来 `WebhookNotifier`(Discord/Slack)を1クラス追加するだけで拡張できる構造にすること(今回は実装しない)

### 4.14 生成スタジオ(画像・動画生成)

テキストプロンプトから画像・動画を生成する専用画面。サイドバーに「生成スタジオ」として追加する。

- **入力部**:プロンプト入力欄+種別選択(画像/動画)+モデル選択ドロップダウン(生成対応モデルのみ表示)+生成ボタン
- **出力部**:生成結果のギャラリー表示(サムネイル一覧+クリックで拡大)。各結果に「保存」「背景に設定」「再生成」「プロンプトをコピー」の操作を付ける
- 生成履歴はプロンプト・モデル・日時とともに保存し、後から同じプロンプトで再生成できるようにする
- 「背景に設定」は §4.12 の背景カスタマイズと連動する(生成→即背景適用の1タップ動線)
- [確認が必要な点] 画像生成はOpenAI・GoogleのAPIで提供されているが、動画生成APIの提供状況・対応モデル・料金・生成時間は実装時に各社最新ドキュメントで確認すること。動画生成は高コスト・長時間になる場合があるため、生成前に概算コストと目安時間を表示する確認ダイアログを必ず挟む
- 生成中は進行状態を表示し、失敗時はエラー理由を表示する(非同期処理。生成中も他画面の操作を妨げない)

### 4.15 APIキーの入力フロー

- **初回起動時**:セットアップウィザードを表示し、各プロバイダのキー入力欄を提示する(スキップ可能。キーの発行場所への案内リンクを添える)
- **設定画面**:サイドバー「設定」内にキー管理セクションを置き、いつでも追加・変更・削除できる。キーは伏せ字表示(末尾4文字のみ表示)とする
- キーはOSキーチェーンに保存(§9参照)。同期対象外のため、**2台のPCそれぞれで初回に入力が必要**である旨をウィザード内に明記する
- キー未入力のプロバイダは、役割カード作成画面・生成スタジオのモデル選択でグレーアウト表示し、クリック時に設定画面へ誘導する

### 4.16 確認質問プロトコル(クラリフィケーション)

役割カードが、情報不足・曖昧さを推測で埋めずにユーザーへ聞き返す仕組み。

- 全役割カードのシステムプロンプトに共通規約を自動注入する:「実行に必要な情報が不足または曖昧な場合、推測で補完せず、次のJSON形式のみで質問を返すこと」
- 質問JSONの形式(例):

```jsonc
{
  "type": "clarification",
  "questions": [
    {
      "text": "視点はどちらで始めますか?",
      "options": ["ノクト側", "虎杖側"],   // 選択肢(任意)
      "allow_free_text": true              // 自由入力の可否
    }
  ]
}
```

- アプリ側はレスポンスがこの形式か判定し、該当時は**質問カードUI**(選択肢ボタン+自由入力欄)として表示する。ユーザーの回答を会話に注入して実行を継続する
- パース失敗時・非該当時は通常のテキスト返答として表示する(フェイルセーフ)
- **パイプライン実行中の挙動**:ステップ実行中に質問が返された場合、パイプラインを一時停止して質問カードを表示し、回答後に該当ステップから再開する。パイプライン定義に `allow_clarification: true|false` を持たせ、無停止で完走させたい場合はオフにできる(オフ時は「不明点は保留事項として出力末尾に列挙せよ」という指示に差し替える)
- 質問とその回答も通常のメッセージとして会話ツリーに記録する(観測可能性の確保)

## 5. デザイン仕様

管理画面(ダッシュボード)型レイアウト。参考にした既存テンプレートの直接コピーは避け、以下の確定仕様に従う。

### 5.1 レイアウト骨格

- 左:折りたたみ可能なサイドバーナビ。項目:ホーム/チャット/役割カード/分岐/タスク/**生成スタジオ**/コスト/設定。「一般」「分析」等のセクション見出しで分類
- 上:ヘッダーバー(検索・通知・ダーク切替)
- 中央:カードグリッド

### 5.2 配色(確定値はデザイントークンとして定義)

- **ライトモード**:サイドバー=濃紺グレー(基準 #2A3F54)、選択中項目=左に緑(#1ABB9C)の3pxボーダー+一段暗い背景。コンテンツ背景=薄グレー(#F4F6F8)、カード=白・**枠線と影なしのフラット**
- **ダークモード**:純黒ではなく暗い青グレー基調(ページ #15191E、カード #1E252E)。アクセント緑は共通。長時間作業での目の疲労を避けるため純黒は使わない
- 統計カード:数値+増減表示(例:「3 ↑1 今週」)

### 5.3 カード内部

- 見出し行+**太めの下線(2px)**で見出しと本文を区切る(全カード共通)
- 見出し行の右端にメタ情報(担当AIバッジ、並び順など)を配置
- リスト行間の区切りは細線(1px)。見出し線との太さの差で階層を表現する

### 5.4 タスクの期日表示

- プログレスバーは使わない。**日付テキスト+相対表現**(例:「7/9(明日)」「7/12(日)」)
- 緊急度は文字色のみで表現:期限が翌日以内=赤(#E74C3C)、それ以外=グレー

### 5.5 AIの識別

- **枠色・背景色による色分けはしない**。グレーの小バッジ(例:「秘書役 · Claude」)のみで示す
- `RoleCard.color` はデータとして保持し、設定で色表示をオンにできる余地を残す(既定オフ)

## 6. データ構造

### 6.1 ワークスペース(同期フォルダ)構成

```
ワークスペース/(Google Drive等の同期フォルダ内)
├── settings.json            … アプリ設定・テーマ
├── role_cards/{card_id}.json
├── shared_memory/
│   ├── index.json
│   └── {doc_id}.md
├── conversations/
│   ├── index.json
│   └── {conv_id}.json       … メッセージツリー含む
├── pipelines/{pipeline_id}.json
├── tasks.json
├── media/
│   ├── index.json           … 生成履歴(プロンプト・モデル・日時)
│   └── {media_id}.(png|mp4) … 生成した画像・動画の実体
└── usage/{YYYY-MM}.json     … トークン記録(月別)
```

分割方針の理由:更新頻度の高いデータ(会話)を1件1ファイルにすることで、2台同時使用時の同期衝突を「同じ会話を同時編集した場合」だけに限定する。usageを月別にするのは単一ファイルの肥大化と全体転送を避けるため。

media/ の注意:動画ファイルは容量が大きくなりやすい。設定に「動画を同期対象に含める/ローカルのみに保存」の切り替えを設け、既定は「画像=同期、動画=ローカルのみ」とする(クラウドストレージの容量圧迫と同期の遅延を避けるため)。

### 6.2 エンティティ定義

```jsonc
// role_cards/{card_id}.json
{
  "id": "rc_001",
  "name": "執筆担当",
  "provider": "anthropic",            // anthropic | openai | google
  "model": "(モデル一覧APIから選択したID)",
  "system_prompt": "…",
  "shared_memory_refs": ["sm_bible01"],
  "color": null,                       // 既定null=色表示オフ
  "tools": { "web_fetch": false, "folder_scan": false },
  "triggers": [],                      // 例: ["on_folder_scan", "on_app_start"]
  "created_at": "", "updated_at": ""
}
```

```jsonc
// conversations/{conv_id}.json
{
  "id": "conv_012",
  "title": "第12話の執筆",
  "active_leaf_id": "msg_045",         // 本線の最先端メッセージID
  "messages": [
    {
      "id": "msg_044",
      "parent_id": "msg_043",          // null=会話の起点
      "author": "user",                // user | assistant
      "role_card_id": null,            // assistant時のみ
      "model_override": null,          // 一時切り替え時のみ
      "content": "…",
      "tokens": { "input": 1200, "output": 900 },
      "created_at": ""
    }
  ]
}
```

- 本線の表示:`active_leaf_id` から `parent_id` を根まで辿って復元
- 分岐の本線切り替え:`active_leaf_id` の書き換えのみ。メッセージ本体は移動・複製しない

### 6.3 同期対象外(各PCローカル)

- APIキー:OSキーチェーンに保存。各PCで初回に個別入力する
- キャッシュ・一時ファイル

## 7. 開発フェーズ

| フェーズ | 内容 |
|---|---|
| **1(コア)** | チャット+役割カード+@メンション振り分け+モデル切り替え(2段階)+共有バイブル注入+**トークン記録**+メッセージのツリー構造データ+ライト/ダークテーマ基盤 |
| **2(執筆ワークフロー)** | パイプライン+ログ保存+会話分岐UI(比較・本線切替)+背景カスタマイズ(持ち込み)+情報収集カード+**確認質問プロトコル(§4.16)** |
| **3(学生機能)** | 締め切り検知(スキャン+確認UI+タスク欄)+試験対策パイプライン+朝ブリーフィング+OS通知(NotificationSink構造)+AI生成背景 |
| **4(仕上げ)** | 2台同期の最終確認+コストダッシュボード+役割仕分けウィザード+**生成スタジオ(画像→動画の順に対応)** |

生成スタジオをフェーズ4に置く理由:画像・動画生成はチャット基盤と独立した機能であり、コア価値(執筆ワークフロー・タスク管理)の完成を優先するため。ただし §4.15 のAPIキー入力フローはフェーズ1で実装する(チャット機能の前提となるため)。

フェーズ1完了時点でユーザーの小説執筆ワークフローに実戦投入できることを最優先とする。

## 8. トークン節約設計

API利用コストを抑えるための設計要件。特に 8.1 と 8.3 は後付けが困難なため、フェーズ1の実装時点から適用すること。

### 8.1 プロンプトキャッシュ(最優先)

- 各社のプロンプトキャッシュ機構(Anthropic: prompt caching / Google: context caching / OpenAI: 自動キャッシュ)を利用する
- **API呼び出しの構成順序を固定する**:先頭から「①システムプロンプト(役割定義+共通規約)→②共有バイブル→③会話履歴→④最新の入力」の順に組み立てる。①②は毎回同一内容になるよう正規化し(余分な空白・タイムスタンプ等を含めない)、キャッシュの先頭一致に乗せる
- [確認が必要な点] 各社のキャッシュの指定方法(明示的なcache_controlか自動か)、最小キャッシュサイズ、有効期間、割引率は実装時に最新ドキュメントで確認すること

### 8.2 会話履歴の窓+要約圧縮

- 会話履歴は全件送信せず、**直近N往復(既定10、設定で変更可)を原文のまま**、それより古い部分は**要約に圧縮して**送信する
- 要約の生成は軽量モデル(Flash / Haiku / mini系)を使用し、会話ファイル内にキャッシュとして保存する(毎回作り直さない)
- 会話ツリーとの整合:本線(active_leaf_idから根まで)を対象に「古い区間の要約+直近区間の原文」を構成する
- 要約の注意:共有バイブル(§4.3)と重複する情報は要約から省く

### 8.3 コードの差分出力規約

- 全役割カードの共通規約に以下を注入する:「既存のコード・文書の修正を指示された場合、全文を再出力せず、変更箇所のみを『変更前→変更後』の形式で示すこと。全文が必要な場合はユーザーが明示的に要求する」
- 出力トークンは入力より単価が高いため、修正系のやり取りで特に効果が大きい

### 8.4 タスク別モデルルーティング

- 精度より量のタスクは軽量モデルを既定にする。具体的には:締め切り検知(§4.6)の抽出、朝ブリーフィング(§4.8)の生成、履歴要約(§8.2)、試験対策パイプラインの一次要約
- 役割仕分けウィザード(§4.10)が生成するカードにも、タスク性質に応じた軽量/上位モデルの推奨を含める

### 8.5 その他の節約策

- 全API呼び出しに `max_tokens` 上限を設定する(用途別の既定値を設け、設定で変更可)
- 共有バイブルは資料ごとに「全文/要約版」を持てるようにし、カード側でどちらを注入するか選択できるようにする(要約版はアプリ内で生成・保存)
- 急がない一括処理(試験対策パイプライン等)には各社のBatch APIの利用を検討する([確認が必要な点] 対応モデル・割引率は要確認)
- コストダッシュボード(§4.11)にキャッシュヒットによる節約分も表示できるとなお良い(各社usageにキャッシュ利用量が含まれる場合)

## 9. セキュリティ・安全要件

1. APIキーは同期フォルダ・設定JSONに平文で保存しない(OSキーチェーンのみ)
2. Web取得はユーザーが明示したURLに限定。自動クロールは実装しない
3. 締め切り検知は必ずユーザー確認を挟んでからタスク登録する
4. 外部送信はユーザーのAPIキーによる各社公式APIへのリクエストのみ。それ以外の外部通信は行わない

## 10. 実装時に確認が必要な点(まとめ)

- 各社API利用規約の最新版(個人アプリでの統合利用条件)
- 各社モデル一覧エンドポイントのレスポンス形式とチャット対応モデルのフィルタ方法
- 各社APIのトークン使用量(usage)の返却形式と正規化方法
- 画像生成APIの最新の提供状況・モデル名
- 動画生成APIの提供状況・対応モデル・料金体系・生成所要時間・出力形式
- Tauriでのフォルダ監視・OS通知・キーチェーンアクセスの各プラグインの最新版
- 各社プロンプトキャッシュの指定方法・最小サイズ・有効期間・割引率(§8.1)
- 各社Batch APIの対応モデル・割引率・所要時間(§8.5)
